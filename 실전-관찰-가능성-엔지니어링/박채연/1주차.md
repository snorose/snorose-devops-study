## 2. 핵심 내용 요약

전통적인 모니터링은 미리 정의한 지표와 조건을 기준으로 이미 알고 있는 문제를 탐지하는 데 효과적이다.
하지만 분산 시스템에서는 예상하지 못한 문제가 발생하며, 기존 대시보드와 알람만으로 원인을 찾기 어렵다.
관찰 가능성은 로그, 메트릭, 트레이스를 연결해 새로운 질문을 던지고 시스템의 내부 상태를 추론할 수 있게 한다.
현재 시스템도 여러 도구를 오가며 사람이 직접 상관관계를 파악해야 하므로, 하나의 문맥에서 탐색할 수 있는 구조가 필요하다.

## 3. 주요 개념

| 개념     | 설명                                                     | 현재 시스템과의 관계                                   |
| ------ | ------------------------------------------------------ | --------------------------------------------- |
| 모니터링   | 사전에 정의한 지표와 임계값으로 알려진 문제를 탐지하는 방식이다.                   | 현재는 오류 로그와 인프라 메트릭을 기준으로 장애를 감지하고 있다.         |
| 관찰 가능성 | 시스템이 남긴 데이터를 통해 예상하지 못한 문제의 원인을 탐색하고 내부 상태를 추론하는 능력이다. | 로그, 메트릭, 배포 이력을 연결해 한 문맥에서 분석할 수 있는 구조가 필요하다. |

## 4. 인상 깊었던 내용

> 시스템의 상태에 대해 무엇을 알아낼 수 있는지를 확인함으로써 신경 쓰고 있는 조건을 반복적으로 조사할 수 있다.

분산 시스템의 예상하지 못한 장애는 같은 형태로 반복되지 않기 때문에, 모든 문제에 대응하는 대시보드를 미리 만드는 것은 어렵다. 결국 이미 알고 있는 문제를 감시하는 것보다, 알지 못하는 문제를 탐색할 수 있는 구조를 만드는 것이 중요하다.

스테이징 환경은 프로덕션과 비슷하게 구성할 수는 있지만 실제 트래픽과 데이터, 구성 요소 간 상호작용까지 완전히 재현할 수 없는 ‘무균실’에 가깝다는 점도 인상적이었다. 따라서 프로덕션에서 발생하는 실제 데이터를 안전하게 관찰할 수 있어야 한다.

## 5. 이해하기 어려웠던 내용

> 디버거는 더 이상 특정한 런타임에 연결할 수 없다.

분산 시스템에서는 하나의 요청이 여러 서버와 서비스, 네트워크 구간을 거친다. 따라서 특정 프로세스에 디버거를 연결하는 것만으로는 전체 요청 흐름을 확인할 수 없다.

즉 디버거를 사용할 수 없다는 뜻이 아니라, 하나의 런타임만 확인하는 기존 방식으로는 전체 문제를 추적하기 어렵다는 의미다. 이 때문에 로그, 메트릭, 트레이스를 연결해 요청 흐름을 재구성해야 한다.

> 처음에는 카오스 엔지니어링을 하려면 시스템의 모든 동작을 완전히 이해하고 있어야 한다는 뜻처럼 보여 이해하기 어려웠다.

하지만 핵심은 전체 구조를 모두 아는 것이 아니라, 현재 시스템의 정상 상태를 판단할 기준이 있어야 한다는 의미였다. 정상 상태를 정의한 뒤 일부러 장애를 발생시키고, 시스템이 어떻게 흔들리며 다시 복구되는지를 로그와 메트릭으로 확인하는 것이다.

즉 카오스 엔지니어링은 단순히 장애를 일으키는 실험이 아니라, 장애 상황에서도 시스템이 예상한 방식으로 동작하는지를 검증하는 과정이라고 이해했다.
