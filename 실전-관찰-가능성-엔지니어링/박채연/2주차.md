## 2. 핵심 내용 요약
5장부터 9장에서는 앞서 추상적으로 설명했던 관찰 가능성이 실제로 어떻게 구현되는지 간단한 사례를 통해 보여준다. 특히 기존 모니터링의 주요 요소인 메트릭과 로그만으로는 관찰 가능성을 충분히 구현하기 어려운 이유를 설명하고, 계측과 분산 추적을 통해 데이터를 수집하고 분석하는 방식을 소개한다.

이를 통해 모니터링과 관찰 가능성의 차이를 보다 분명하게 이해할 수 있었다. 모니터링이 운영자의 경험과 직관을 바탕으로 가설을 세우고 검증하거나 제거하는 과정이라면, 관찰 가능성은 실제로 수집된 증거를 바탕으로 가능성 높은 가설을 세운 뒤, 데이터를 다양한 디멘션으로 탐색하며 실제 원인으로 범위를 좁혀가는 과정에 가깝다.

앞부분에서는 모니터링을 다소 낡은 방식처럼 구분했지만, 실제로 관찰 가능성이 모니터링을 완전히 대체하는 것은 아니다. 모니터링으로 이미 알려진 장애를 빠르게 감지하고, 관찰 가능성을 통해 원인을 깊이 분석하는 방식으로 두 접근법을 함께 활용해야 한다는 점을 알 수 있었다.


## 3. 주요 개념

| 개념     | 설명                                                     | 현재 시스템과의 관계                                   |
| ------ | ------------------------------------------------------ | --------------------------------------------- |
| 이벤트   |  시스템에서 발생한 개별 동작이나 상태 변화                |   현재 Spring Boot 애플리케이션 로그와 systemd 로그에 여러 이벤트가 기록되고 있다. 다만 대부분 문자열 로그 형태라서 request_id, user_id, file_id, status 같은 공통 필드로 구조화되어 있지 않으면 이벤트 간 관계를 찾기 어렵다.    |
| 추적(Trace) | 서로 연관된 일련의 이벤트 집합 ,하나의 작업이 여러 단계로 처리되는 전체흐름을 연결해서 보는 방식 , 각 작업을 "스팬"이라고하고, 여러 스팬을 하나의 "트레이스"로 묶는다. | 현재는 Loki 로그를 통해 개별 오류를 확인할 수 있지만, 하나의 요청이 어떤 단계를 거쳤고 어느 구간에서 지연되었는지를 자동으로 연결해서 보기는 어렵다. OpenTelemetry와 Tempo 또는 Zipkin을 도입하면 요청 단위의 처리 흐름과 병목 구간을 확인할 수 있다. |
| 계측 | 애플리케이션이나 인프라에서 로그, 메트릭, 트레이스와 같은 관찰 데이터를 생성하도록 하는 과정. 라이브러리를 이용하면 자동 계측, 직접 코드 구간으로 구현하면, 수동 계측 |  현재 Spring Actuator와 Micrometer는 메트릭 계측, Logback은 로그 계측에 해당한다. 다만 트레이스 계측은 아직 없으므로 OpenTelemetry Java Agent를 적용하거나, 중요한 비즈니스 로직에 직접 스팬과 사용자 정의 속성을 추가하는 작업이 필요하다.
|  가설 주도 디버깅    | 장애 원인으로 의심되는 가설을 세우고, 수집된 데이터를 이용해 이를 검증하거나 제거하는 방식 |현재도 로그와 메트릭을 보며 원인을 추측하고 있지만, 데이터가 서로 연결되어 있지 않아 가설 검증에 시간이 오래 걸릴 수 있다. Request_id를 기준으로 연결해 보자. 이것도 부족하면, Trace Id 를 도입해보자
| 핵심 분석 루프 | 이상 징후를 발견한 뒤 관련 이벤트를 찾고, 데이터를 여러 디멘션으로 분해·비교하면서 원인을 좁힌 다음, 새로운 가설을 세워 다시 검증하는 반복 과정이다. | 현재는 Grafana 대시보드와 Loki 검색을 통해 일부 분석이 가능하지만, 미리 정의된 패널과 라벨에 의존하는 한계가 있다. 서비스, 엔드포인트, 사용자, 상태 코드, 파일 크기 등의 디멘션을 추가하면 문제와 연관된 조건을 더 자유롭게 탐색할 수 있다.

## 4. 인상 깊었던 내용
> 상호의존성이 있다면 원인이 모호해지는 경우가 많은데, 추적을 이용하면 시스템간 의존성을 더 잘 이해할 수 있다. ->  디버깅에 도움 많이 된다.

여러 서비스나 외부 시스템이 연결된 환경에서는 장애가 발생한 위치와 실제 원인이 다를 수 있다. 분산 추적을 사용하면 요청이 어떤 서비스와 외부 의존성을 거쳤는지 확인할 수 있어 디버깅에 큰 도움이 될 것 같다.

> 문제 코드 블록을 하나의 개별 스팬으로 감싸서 더 자세한 워터폴 시각화를 확보한다

하나의 Trace 안에 여러 개의 Span을 생성하면 요청 전체 흐름뿐만 아니라 애플리케이션 내부의 세부 처리 시간까지 확인할 수 있다. 예를 들어 파일 업로드 요청을 다음과 같이 나눌 수 있다.

Trace: 파일 업로드 처리
├─ Span: HTTP 요청 수신
├─ Span: 파일 검증
├─ Span: 메타데이터 JSON 역직렬화
├─ Span: S3 업로드
└─ Span: DB 저장
각 코드 구간을 별도 스팬으로 만들면, JSON 역직렬화가 CPU를 많이 사용하며 병목을 만든다는 걸 확인할 수 있다

> 관찰 가능성의 진정한 힘은 실제 비즈니스 로직 동작을 디버깅 할 수 잇도록 custom attributes을 활용한 context 정보가 추가되었을 때 발휘된다. 

Trace와 Span에 user_id, file_size, endpoint, status, retry_count 같은 사용자 정의 속성을 추가하면 기술적인 처리 시간뿐만 아니라 어떤 조건에서 문제가 발생했는지도 분석할 수 있다. 단순한 시스템 지표를 넘어 실제 비즈니스 로직의 동작을 분석할 수 있다는 점이 인상 깊었다.

-> cardinality에 따른 성능 부하 영향은 안받나..?


>관찰 가능한 시스템에서는 서로 다른 곳에서 일어난 이벤트들을 하나로 엮을 수 있어야한다. 

현재 진행 중인 로그 구조화 작업과 직접적으로 연결되는 부분이라고 느꼈다. 여러 줄로 출력되는 비정형 로그를 구조화하고, 공통 필드를 정의하며, request_id나 trace_id를 추가하는 작업은 서로 다른 이벤트를 하나의 요청 흐름으로 연결하기 위한 기초가 된다.


## 5. 이해하기 어려웠던 내용

> 처음에는 계측과 추적이 비슷한 개념처럼 느껴졌다

그러나 계측은 데이터를 만들어내는 과정이고, 추적은 그 결과로 생성된 스팬들을 하나의 요청 흐름으로 연결하여 분석하는 방식이라고 이해했다.
애플리케이션 코드 
↓ 계측 
Span, Event, Attribute 생성 
↓ 연결 
Trace 구성 
↓ 저장 및 시각화 
Tempo 등에서 분석


> 분산 추적의 데이터 전달 방식

분산 추적은 여러 서비스를 횡단하는 하나의 요청을 연결하기 위해 최초 요청에서 고유한 Trace ID를 생성한다. 이후 서비스 간 HTTP 요청을 보낼 때 Trace ID와 현재 Span의 정보를 헤더에 담아 전달한다

각 서비스는 전달받은 컨텍스트를 기반으로 새로운 Span을 만들고, 이를 동일한 Trace에 포함시킨다. 각 Span에는 시작 시각, 실행 시간, 상태, 사용자 정의 속성 등이 기록된다.

이를 통해 요청이 어떤 경로를 거쳤는지, 어느 구간이 오래 걸렸는지, 어떤 조건에서 오류가 발생했는지를 하나의 Trace에서 확인할 수 있다.

## 6. 한줄평

과거 React로 대시보드를 구현했을 때, 클라이언트는 미리 정해진 그래프만 보는 것이 아니라 사용자가 직접 데이터와 차트를 구성할 수 있기를 원했다. 당시에는 왜 이렇게 높은 자유도가 필요한지 완전히 이해하지 못했지만, 이번 내용을 읽으며 자유로운 탐색 자체가 새로운 인사이트를 발견하기 위한 중요한 조건일 수 있다는 점을 알게 되었다.

물론 자유도가 높다고 해서 항상 좋은 것은 아니다. 사용자가 도메인 지식이나 분석 기준을 충분히 갖고 있지 않다면 어떤 데이터를 선택하고, 어떤 디멘션으로 나누어 봐야 하는지조차 판단하기 어려울 수 있다. 결국 자유로운 대시보드는 탐색 가능성을 넓혀주지만, 그 가능성을 실제 인사이트로 연결하려면 사용자의 지식과 분석 역량이 필요하다.

책에서 소개한 버블업 분석은 이러한 탐색 과정을 어느 정도 자동화한 방식으로 보였다. 사용자가 모든 디멘션을 직접 하나씩 비교하지 않아도, 문제와 연관성이 높은 디멘션을 계산해 보여준다는 점이 인상적이었다. 완전히 자유로운 대시보드와 자동화된 분석 방식에는 차이가 있지만, 정해진 그래프만 제공하는 것이 아니라 다양한 관점에서 데이터를 탐색할 수 있어야 새로운 인사이트를 발견할 수 있다는 점에서 예전에 구현했던 대시보드가 떠올랐다.
