# 핵심 내용
## 관찰 가능성 vs 모니터링
- 기존 모니터링 방식은 사후 대응적인 접근 방법이다
    - 이전에 발견되었던 오류 조건이 발생했는지를 나타내는 임계치와 비교해서 시스템 상태가 괜찮은지를 확인하는 방식으로 동작하기 때문에.
- 관찰 가능성 도구들은 반복적인 탐색 조사를 통해 성능 문제가 어디서, 왜 발생했는지를 시스템적으로 찾아낼 수 있게 해준다.
    - 관찰 가능성은 이전에 발생했던 문제 여부와 관계없이 장애 모드를 적극적으로 식별할 수 있게 해준다.
        -> 도대체 어떻게 한다는걸까
- 모니터링 데이터를 활용한 디버깅은 많은 경우에 직관에 의존한다.
    - -> 확증편향으로 인해 문제의 진짜 원인 파악을 어렵게 만드는 경향이 있음
- 관찰 가능성을 통한 더 나은 디버깅
    - 어떤 디멘션의 조합에 대해 어디서, 어떻게 문제가 발생할지를 예측할 필요 없이 문제의 근본적 원인을 명시적으로 찾을 수 있도록 해준다.
        - 🙄 도대체 어떻게 그렇게 할 수 있다는건데???

- 제도적 지식: 문서화되지 않은 정보로 조직 내에서 일부 인원들에게 알려졌지만 대부분은 잘 알지 못하는 지식
    - 팀에서 가장 오래 근무한 엔지니어가 문제 해결을 위한 최고의 디버거이자 최후의 디버거가 되는... 모니터링 기반 접근방법 사용하는 팀...

- 반대로! 관찰가능성을 활용하는 팀 내의 최고 디버거는 가장 호기심 많은 엔지니어다.

- 🫩 저 어떻게에 대해 언급하겠다면서 아직 여전히 추상적이고 모호한 이야기 뿐이다.
