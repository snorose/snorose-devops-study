# 핵심 내용
- `관찰 가능성 != 단순 로그, 메트릭, 트레이스` 라는 관점을 전달하고자 한 파트인 것 같다.
- 소프트웨어 시스템에 대한 관찰 가능성의 정의: 시스템의 상태가 얼마나 새롭고 이상한지와 관계없이 그 상태를 얼마나 잘 이해하고 설명할 수 있는지를 측정하는 척도
    - 여기 번역된 문장이 너무 복잡해서 아직 모호하게 느껴진다.
    - 관찰 가능성을 확보하는 것 = 추가적인 코드 작성 없이도 시스템이 가질 수 있는 상태에 대해 이해하고 설명할 수 있게 된다는 것
- 카디널리티
    - 높은 카디널리티가 좋다.
    - 카디널리티란?: 한 집합에 포함된 데이터 값의 고유성.
        - e.g. 성별은 카디널리티가 낮지만, id값 같은 PK는 카디널리티가 매우 높다.
    - 그렇기 때문에 카디널리티가 높은 정보는 디버깅에 필요한 데이터를 식별하거나, 시스템을 이해하고자할 때 가장 유용하다.
    - 그래서 관찰 가능성 관점에서 카디널리티가 매우 중요함.
    - 메트릭 기반의 도구들은 낮은 카디널리티를 갖는 디멘션만 다룰 수 있다.는데... 이해하지 못함.
        - "비교할 호스트가 수백 개에 불과하더라도 메트릭 기반의 시스템에서는 카디널리티에 대한 키 공간의 한계에 다다르지 않고서는 호스트명을 식별을 위한 태그로 사용하기 어렵다."
        - 이 문장을 풀어서 이해해보자.
            - -> 메트릭 시스템 내부에서 메트릭명 + 태그 조합을 하나의 고유 key처럼 관리하는데,
            hostname, request_id 처럼 카디널리티가 높은 식별자를 태그로 사용하면, 이 고유 key가 너무 많아져서 시스템이 감당 가능한 범위를 넘게 된다(저장량·메모리·쿼리 비용이 폭발)
            - -> 그래서 낮은 카디널리티를 갖는 디멘션만 태그로 활용하게 되는 경우가 많다는 의미.

- 디멘셔널리티
    - 이 역시 높을수록 좋다.
    - 디멘셔널리티란?: 메트릭 하나에 붙어 있는 디멘션의 개수
    - 참고: 디멘션 = 태그 = 라벨이라고 보면 됨
        - 디멘션은 메트릭을 나눠서 보기 위한 기준값이다.
        - 예를 들어: http_requests_total{env="prod", status="500", service="api"} 에서
            - env / status / service가 디멘션
            - 그리고 prod / 500 / api가 디멘션 값
    - 디멘셔널리티가 높을수록, 애플리케이션 동작에 숨겨져 있거나 파악하기 어려운 패턴을 찾을 수 있는 가능성이 높아진다.
